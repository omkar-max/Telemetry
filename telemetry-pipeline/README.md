Telemetry Pipeline – Design Notes

1. Motivation

The purpose of this project is to implement a simple telemetry processing pipeline using Linux systems programming concepts. The system simulates multiple producer processes generating telemetry frames and a central aggregator that consumes and processes those frames concurrently.

The main objective of the project was to practice and demonstrate the following concepts:

- POSIX shared memory IPC
- POSIX semaphores for synchronization
- Ring buffer data structures
- Multi-threaded processing using pthreads
- Producer–consumer pipeline architecture
- Basic runtime monitoring and structured logging

The system is intentionally designed in stages so that frame ingestion, processing, and logging are separated.

---------------------------------------------------------------------

2. High Level Architecture

The pipeline consists of two types of processes:

1) Producer processes  
2) Aggregator process

Producers generate telemetry frames and write them into a shared memory ring buffer. The aggregator reads frames from this buffer and distributes them to worker threads for processing.

High level data flow:

    Producers
        |
        v
    Shared Memory Ring Buffer
        |
        v
    Aggregator Reader Thread
        |
        v
    Internal Queue
        |
        +---- Worker Thread 0
        +---- Worker Thread 1
        +---- Worker Thread N
        |
        v
    CSV Logging + Runtime Statistics

Each stage is separated so that slow processing does not block data ingestion.

---------------------------------------------------------------------

3. Producer Process

Producer processes simulate telemetry sources.

Each producer performs the following steps:

1) Connect to the shared memory region
2) Wait for a free slot in the ring buffer
3) Write a telemetry frame
4) Update its sequence number
5) Signal the consumer that a frame is available

Example frame structure:

    producer_id = 1
    sequence_number = 101
    payload = "Hello from Producer 1 seq=101"

The sequence number increases for every frame generated by a producer. This allows the aggregator to detect missing frames.

---------------------------------------------------------------------

4. Shared Memory Ring Buffer

The shared memory region contains a circular ring buffer used for communication between producers and the aggregator.

The buffer header maintains the following fields:

    head   -> index where the next frame will be written
    tail   -> index where the next frame will be read
    count  -> number of frames currently stored
    cap    -> total capacity of the buffer
    drops  -> counter for overwritten frames

Example layout:

    index: 0   1   2   3   4
          [F] [F] [ ] [ ] [ ]

    tail -> next frame to read
    head -> next frame to write

When the end of the array is reached, the buffer wraps around to the beginning.

Using a ring buffer keeps memory usage constant and avoids dynamic allocations.

---------------------------------------------------------------------

5. Synchronization

Three semaphores are used to coordinate access to the shared memory buffer.

empty semaphore

Tracks the number of free slots in the buffer.

Initial value: buffer capacity

A producer must wait on this semaphore before writing a frame.

---------------------------------------------------------------------

full semaphore

Tracks how many frames are available for the aggregator to read.

Initial value: 0

The aggregator waits on this semaphore before consuming frames.

---------------------------------------------------------------------

mutex semaphore

Ensures only one process modifies the buffer metadata at a time.

The following fields are protected:

    head
    tail
    count

This prevents race conditions when multiple producers are writing simultaneously.

---------------------------------------------------------------------

6. Aggregator Process

The aggregator is responsible for consuming frames from shared memory and processing them.

Instead of using a single thread, the aggregator uses multiple threads:

    Reader thread
    Worker threads
    Statistics thread

Each thread performs a specific role in the pipeline.

---------------------------------------------------------------------

7. Reader Thread

The reader thread continuously reads frames from the shared memory ring buffer.

Steps performed by the reader thread:

1) Wait on the "full" semaphore
2) Acquire the mutex protecting the buffer
3) Read the frame at the tail position
4) Update tail and count
5) Release the mutex
6) Signal the "empty" semaphore
7) Push the frame into the internal queue

The reader thread performs minimal work so that it can keep up with producers.

---------------------------------------------------------------------

8. Internal Queue

The aggregator maintains an internal queue between the reader thread and the worker threads.

This queue decouples frame ingestion from frame processing.

Without this queue, worker threads would have to access shared memory directly, increasing synchronization complexity.

Benefits of this design:

- reduces contention on shared memory
- allows multiple workers to process frames concurrently
- improves throughput

The queue is protected using:

    pthread mutex
    pthread condition variables

---------------------------------------------------------------------

9. Worker Threads

Worker threads perform the actual frame processing.

Each worker thread runs a loop that:

1) Pops a frame from the internal queue
2) Processes the frame
3) Writes a structured log entry
4) Updates runtime statistics

Using multiple workers allows the system to process frames in parallel.

---------------------------------------------------------------------

10. Structured Logging

Processed frames are written to a CSV file.

Example log output:

    producer_id,sequence_number,worker_id,payload
    1,101,0,Hello from Producer 1 seq=101
    2,88,1,Hello from Producer 2 seq=88
    1,102,0,Hello from Producer 1 seq=102

A mutex protects the log file so multiple worker threads can write safely.

The CSV format was chosen because it is simple and easy to analyze using tools like Excel or Python.

---------------------------------------------------------------------

11. Runtime Statistics

A statistics thread periodically prints system metrics.

Example:

    received_total   : 2480
    processed_total  : 2480
    throughput       : 620 frames/sec
    queue_depth      : 5 / 512
    shared_drops     : 0
    seq_gaps_detected: 0

These metrics help monitor system behavior and detect issues such as:

- dropped frames
- queue congestion
- uneven worker utilization

---------------------------------------------------------------------

12. Sequence Gap Detection

Each producer generates frames with monotonically increasing sequence numbers.

If the aggregator receives:

    last_seq = 100
    current_seq = 105

then frames 101–104 were missed.

Possible causes include:

- ring buffer overwrites
- producers running faster than consumers
- synchronization errors

The system tracks these events for debugging purposes.

14. Limitations

This implementation is primarily intended as a learning exercise.

Possible improvements include:

- graceful shutdown support
- configurable buffer sizes
- JSON structured logging
- performance benchmarking
- network-based producers

---------------------------------------------------------------------

15. Summary

This project demonstrates a simple telemetry processing pipeline using Linux systems programming primitives.

Key concepts demonstrated include:

- shared memory IPC
- semaphore synchronization
- circular buffer data structures
- multi-threaded processing
- runtime monitoring and structured logging

The architecture follows a producer–consumer pipeline that separates ingestion, processing, and logging stages.
